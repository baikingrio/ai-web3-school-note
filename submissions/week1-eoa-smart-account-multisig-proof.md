# WCB 提交证明：Week 1｜Web3 向进阶｜比较 EOA、智能账户、多签的权限差异

## 建议提交证明

这次我整理了 EOA、智能账户和多签账户的权限差异。我的理解是：这三类账户最大的区别，不只是“地址长得一不一样”，而是**谁真正有控制权、交易怎么被批准、出问题时风险由谁承担**。

## 1. EOA：最普通的钱包账户

EOA 就是我们平时最常见的钱包账户，比如 MetaMask 里直接由助记词 / 私钥控制的账户。

它的特点很直接：

- 谁拿着私钥，谁就能控制账户；
- 发交易、签名、授权都靠这个私钥；
- 不需要合约逻辑，也不需要其他人批准；
- 操作方便，但安全边界也很简单：私钥丢了，基本就没了。

适用场景：

- 个人日常使用钱包；
- 小额测试网操作；
- 学习签名、交易、Gas、合约交互这些基础概念。

主要风险：

- 私钥或助记词一旦泄露，别人就能直接转走资产；
- 用户可能误签恶意授权，比如无限 approve；
- 没有多人确认，也没有内置限额或恢复机制；
- 对 AI Agent 来说，绝对不能把 EOA 私钥交给 Agent。

我的理解：EOA 像一把单人钥匙，方便，但责任和风险都集中在一个人身上。

## 2. 智能账户：带规则的钱包账户

智能账户可以理解成“账户本身是一个智能合约”。它不只是靠一个私钥直接控制，而是可以在合约里写规则。

比如可以设置：

- 某些操作需要额外确认；
- 每天最多花多少钱；
- 只允许调用某些合约；
- 可以设置 session key；
- 可以做恢复、限额、自动化策略；
- 可以让 Agent 在很小的权限范围内帮忙执行。

适用场景：

- 想做更安全的钱包；
- 想给 AI Agent 或脚本一个受限权限；
- 想做自动支付、订阅、游戏钱包、DeFi 策略等；
- 想要比普通 EOA 更细的权限控制。

主要风险：

- 合约代码本身可能有 bug；
- 如果规则配置错了，也可能放大风险；
- 用户可能看不懂复杂的策略；
- 如果自动化权限给得太大，Agent 仍然可能造成损失。

我的理解：智能账户像一把“带规则的钥匙”。它不是谁拿到钥匙就能做所有事，而是要看规则允许什么。

## 3. 多签账户：多人一起管的钱包

多签账户最典型的例子是 Safe。它的核心是：一笔交易不是一个人说了算，而是需要多个 owner 中的一定数量同意。

比如 2/3 多签，就是 3 个 owner 里至少 2 个确认，交易才能执行。

它的特点：

- 控制权分散在多个人手里；
- 可以防止单点私钥失误；
- 更适合团队资金、项目金库、DAO、重要合约管理；
- 高风险操作可以通过多人确认来降低风险。

适用场景：

- 团队金库；
- 项目运营资金；
- 合约 owner 权限管理；
- Hackathon 项目或 DAO 的共同账户。

主要风险：

- 多签 owner 如果管理不好，也可能一起被钓鱼；
- 确认流程比 EOA 慢；
- 如果签名门槛设置不合理，可能太松或太难执行；
- 多签不等于绝对安全，签之前还是要看清楚交易内容。

我的理解：多签像公司账户，不应该一个人随便转账，需要几个人一起确认。

## 4. 三者的关键区别

从“谁控制账户”来看：

- EOA：一个私钥控制；
- 智能账户：按合约规则控制；
- 多签账户：多个 owner 按门槛共同控制。

从“谁可以发起交易”来看：

- EOA：私钥持有人直接发；
- 智能账户：可以由 owner、session key、module、Agent 或其他授权角色发起，但要符合规则；
- 多签账户：可以有人提议交易，但执行通常需要达到确认门槛。

从“是否支持多人确认”来看：

- EOA：不支持；
- 智能账户：可以通过合约逻辑支持；
- 多签账户：天然支持。

从“是否支持限额、恢复或自动化策略”来看：

- EOA：基本不支持，主要靠钱包前端提醒；
- 智能账户：比较适合做限额、恢复、session key、自动化策略；
- 多签账户：适合团队确认，也可以结合 module 做更细的权限控制。

从“风险承担”来看：

- EOA：主要由私钥持有人承担；
- 智能账户：由规则设计者、使用者和合约安全一起承担；
- 多签账户：由多个 owner 共同承担，但每个 owner 仍要认真检查交易。

## 5. 哪些操作必须人工确认

我认为以下操作不应该交给 AI 或脚本自动完成，必须人工确认：

1. 导出、复制、输入私钥或助记词；
2. 主网真实资金转账；
3. 无限 approve 或大额授权；
4. 添加新的收款地址到白名单；
5. 提高每日额度或单笔额度；
6. 修改 Safe owner 或多签门槛；
7. 启用 / 禁用 module；
8. 合约升级、治理投票、团队金库操作。

如果是 AI Agent 参与 Web3 操作，我觉得更安全的方式是：

- 让 Agent 负责解释、检查、生成计划、做 simulation；
- 低风险测试网、小额、白名单内的动作可以自动执行；
- 高风险动作必须人确认；
- 违反规则的动作直接拒绝，而不是让用户“再确认一次就放行”。

## 6. 和我的 AgentScoope Wallet 项目的关系

这次比较也帮助我理解了自己的 Hackathon 项目 AgentScoope Wallet 为什么要用 Safe / Zodiac Roles 这类方案。

如果直接把 EOA 私钥给 Agent，风险太大；如果只让 Agent 给建议，又不能形成完整 workflow。所以我现在的设计是：

- Safe / 智能账户负责承载资产和权限；
- Agent 不是 owner，只是受限 role；
- policy 限制金额、地址和方法；
- simulation 先解释交易结果；
- 高风险操作人工确认；
- 成功或拒绝都写 audit log。

这样 Agent 可以帮忙做低风险执行，但不能随便控制钱包。

## 7. 总结

我的结论是：

- EOA 最简单，适合个人和小额操作，但风险集中在私钥；
- 智能账户更灵活，适合做限额、恢复、自动化和 Agent 权限控制，但要注意合约和策略风险；
- 多签更适合团队和高价值资产，因为它把控制权分散到多人确认。

如果只是学习和小额测试，EOA 很方便；如果要让 Agent 参与链上操作，我更倾向使用智能账户 / Safe / 多签这类可以设置规则和确认流程的账户，而不是把主私钥交给 Agent。
