Problem

Web3 Agent 如果直接拿钱包权限，风险太大

用户不知道 Agent 的边界

最多能花多少钱？能调用哪些协议？能不能换未知 token？失败后怎么追踪？

成功路径之外更需要解释

一个安全的链上 Agent，必须能说明 allowed / denied / pending / failed 每一步为什么发生。

核心问题AI Agent 如何在受限授权边界内执行 DeFi 策略？
Solution

三层安全执行模型

1

Agent proposes

把自然语言收益意图转换成结构化 proposal：预算、资产、协议、期限、风险偏好。

2

Pact decides

用户先看到 Pact Preview：什么允许、什么不允许。proposal 不等于授权。

3

CAW executes

Cobo Agentic Wallet 只在 active Pact 边界内执行；越界动作拒绝并记录日志。

Design principle最小权限 · 可撤销 · 可审计

Implementation

技术栈与执行链路

Frontend

Wallet / Pact

Cobo Agentic Wallet · Pact · Base Sepolia · pact-scoped execution key

Agent / Logs

Hermes runtime · deterministic validation · SQLite / Supabase Postgres · audit trail

关键实现点：active Pact 执行必须使用 Pact 子 Key，不 silent fallback 到 Agent 主 Key；状态与日志需要持久化，才能让执行边界可追踪。

ArchitectureBrowser → Nitro API → DB/Logs → Hermes → Cobo CAW/Pact
Why it matters

项目亮点

Pact-first

先展示授权边界，再展示收益动作。不是 profit-first 的黑盒自动化。

拒绝也是能力

denied path 证明 policy 真的存在，Agent 不能绕过用户授权。

可解释审计

每次 allowed / pending / failed 都能回看原因、状态和日志。

Takeaway安全的 Web3 Agent 应该能说明：能做什么，也不能做什么。
Closing

Agent 可以提出策略，
但不能绕过授权。

YieldAgent 用 Pact 定义边界，用 CAW 受限执行，用日志解释每一步。
