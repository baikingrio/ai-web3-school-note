## 今日学习主题

### PactTrader 的核心定位

今天我把项目主线确定为 PactTrader。它不是一个让 AI 自由交易的工具，而是一个 **policy-bound portfolio rebalancing agent**：

> 用户先定义 token 白名单、协议白名单、单笔上限、日预算、滑点、止损和有效期；Agent 只能在这些 Pact / policy 边界内生成再平衡建议，并通过 CAW 执行层提交受限交易。

这个方向比之前更清楚，因为它把 Hackathon Demo 的重点放在三个问题上：

1. **Agent 可以提出交易建议，但不能绕过授权边界。**
2. **Policy / Pact 负责判断哪些动作允许、拒绝或需要人工确认。**
3. **SQLite 审计日志记录每一步，方便复盘和解释。**

### 今日确定的技术栈

- 前端：Nuxt.js、Vue、TypeScript、Tailwind CSS、shadcn-vue/ui
- 执行层：CAW（Cobo Agentic Wallet）
- Agent / 策略层：Z.AI API
- 数据库 / 日志：SQLite
- 部署：Vercel

我的理解是：

- Nuxt.js 负责把 Demo 页面和 API routes 放在一个轻量全栈项目里；
- Z.AI API 负责生成策略解释或结构化建议，但核心交易边界不能只依赖 LLM；
- 本地 Risk Engine 做确定性检查；
- CAW / Pact 做最终执行边界；
- SQLite 负责保存 proposal、risk decision、execution result 和 audit event；
- Vercel 用来快速部署一个可以展示给评委看的版本。

## Hackathon 推进

### 今日项目理解

PactTrader 的价值不是“AI 帮我炒币”，而是“AI 可以在明确授权的 Pact 边界里做小额、可解释、可撤销、可审计的组合再平衡”。

这让我觉得项目叙事更稳了：

- 普通 AI trading bot 的问题是：Agent 可能幻觉、误判、被诱导，甚至越权交易。
- PactTrader 的做法是：Agent 只生成 proposal，本地 Risk Engine 和 CAW Pact 共同决定能不能执行。
- 即使 CAW 凭证或真实执行环境暂时不可用，也可以用 mock portfolio + dry-run 先展示 allowed / denied / pending / audit log 的完整安全逻辑。

### Demo 优先级

接下来最重要的是先把 Demo 做小：

1. **正常再平衡成功**：当前 ETH 低于目标，Agent 建议 20 USDC 买 WETH，policy pass，进入 dry-run / CAW 执行。
2. **超额交易被拒绝**：Agent 建议金额超过单笔上限，本地 Risk Engine blocked。
3. **未知 token 被拒绝**：Agent 被诱导买非白名单 token，policy blocked。
4. **审计日志可复盘**：每次 proposal、decision、执行结果都能在 dashboard 里看到。
