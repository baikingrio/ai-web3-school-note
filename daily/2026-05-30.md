## 今日学习主题：Agent Profile 与能力声明

今天我把 AgentScoope Wallet Assistant 看成一个有明确身份和边界的 Agent，而不是一个模糊的“AI 钱包助手”。

它的定位是：

> AgentScoope Wallet Assistant 是一个受限钱包执行 Agent，负责把用户的链上操作意图转成可检查、可确认、可审计的执行计划。

这次整理让我更清楚地看到，一个 Agent Profile 至少要回答这些问题：

1. **它是谁**：由谁维护，面向什么用户，服务什么场景。
2. **它能做什么**：意图解析、结构化计划、policy check、simulation、风险分级、审计日志。
3. **它不能做什么**：不能接触私钥、不能绕过白名单、不能自动执行高风险主网操作。
4. **如何被调用**：可以通过 CLI / script / tool calling workflow 接收自然语言任务。
5. **如何验证**：不能只相信 Agent 的文字说明，要结合 tx hash、policy snapshot、Safe / Roles 配置和 audit log。
6. **失败如何处理**：policy 不通过就拒绝，simulation 失败就不广播，高风险动作进入人工确认。

## Hackathon / AgentScoope Wallet 启发

今天这个任务对 AgentScoope Wallet 很有帮助，因为它把项目从“一个钱包 demo”往“一个可以被调用和验证的 Agent 系统”推进了一步。

### 1. Agent 不应该拥有完整钱包权限

Agent 可以理解、计划、解释和调用工具，但不应该直接拥有 Safe owner 权限，也不应该接触私钥、助记词、API Key 或 `.env`。

更合理的方式是：Agent 只是一个受限执行方，所有能力都被 policy、Safe、Zodiac Roles、预算、白名单和人工确认限制住。

### 2. Agent Profile 是项目说明书的一部分

如果未来要让别人理解或调用 AgentScoope Wallet，仅仅说“这是一个 AI 钱包”是不够的。需要说清楚：

- 它的输入是什么；
- 输出是什么；
- 能力边界在哪里；
- 哪些动作会自动执行；
- 哪些动作必须人工确认；
- 失败时如何记录和恢复；
- reviewer 如何验证它不是空泛设计。

### 3. 后续 demo 可以围绕 Profile 展开

后续 Week 3 / Hackathon demo 可以把 Agent Profile 转成实际交互流程：

```text
用户输入链上任务
  → Agent 解析意图
  → 生成结构化计划
  → 检查 policy / budget / allowlist
  → simulation
  → 风险分级
  → 自动执行 / 人工确认 / 拒绝
  → audit log + tx hash + proof
```

这样项目会比单纯展示一笔测试网交易更清楚，因为它展示的是“受限执行系统”的完整边界。
