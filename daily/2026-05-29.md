## 今日学习主题：AI × Web3 最小交叉流程

今天我把一个最小 AI × Web3 工作流拆成了几层：

1. **用户发起任务**：用户用自然语言提出目标，例如“帮我在测试网上支付 0.5 USDC 给白名单服务方”。
2. **AI Agent 生成计划**：AI 不直接控制资产，而是把用户目标转成结构化执行计划。
3. **Policy 检查边界**：检查预算、token、收款方、合约方法和风险等级。
4. **Simulation / 人工确认**：执行前让用户看清楚将要付给谁、付多少、调用哪个合约；高风险动作必须人工确认。
5. **受限钱包执行**：只有在额度内、白名单内、规则通过时，才由受限钱包 / Safe Module 执行。
6. **结果验证**：通过测试网交易哈希、区块浏览器、审计日志和 GitHub proof 验证结果。

我今天最大的收获是：AI × Web3 的重点不是“AI 会不会发交易”，而是要把 **意图理解、权限边界、人工确认、链上执行和结果验证** 串成一个可检查的闭环。

## Hackathon / AgentScoope Wallet 启发

这个任务帮助我把 AgentScoope Wallet 的产品边界讲得更清楚：

### 1. 用户问题

用户希望 AI Agent 可以帮自己完成链上支付、工具购买、赏金发放或小额结算，但又不能把主私钥、无限授权或任意合约调用权限交给 AI。

### 2. 产品答案

AgentScoope Wallet 给 Agent 一个受限执行环境：

- 只能使用指定 token；
- 只能给白名单地址 / 服务付款；
- 有单笔额度和每日预算；
- 执行前要做 simulation；
- 中高风险动作需要人工确认；
- 每次执行、拒绝和原因都留下 audit log。

### 3. Demo 重点

后续 demo 不应该只展示“能付款”，更要展示边界：

- 额度内付款成功；
- 超额付款被拒绝；
- 非白名单收款方被拒绝；
- 高风险动作进入人工确认；
- 所有操作都能回看原因和记录。
