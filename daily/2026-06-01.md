## 今日课程笔记

### 活动主题：从 VC 角度，如何更好打磨项目

今天这场内容我重点关注的不是“怎么讲一个很大的概念”，而是如何把 Hackathon 项目讲得更清楚、更可信、更容易被评委理解。

我准备重点记录这几个问题：

1. **真实问题是否足够具体**
   - AgentScoope Wallet 不是泛泛地说“AI + 钱包”，而是解决一个更具体的问题：AI Agent 可以提出付款 / 采购建议，但不能拿到无限钱包权限。
   - 用户真正担心的是：如果 Agent 误判、被 prompt injection、调用错工具或遇到恶意输入，资金会不会被越权转走。

2. **目标用户是否清楚**
   - 目标用户不是所有 Web3 用户，而是已经在使用 AI Agent / AI coding / research agent 的 builder。
   - 他们希望 Agent 能购买 API、数据、工具服务，或者执行低风险小额付款，但又需要预算、白名单、人工确认和审计记录。

3. **MVP scope 是否足够小**
   - Week 4 不应该做完整 marketplace、escrow、DeFi 策略或多 Agent 协作。
   - 最小 Demo 只需要证明四条路径：额度内成功、超额拒绝、非白名单拒绝、撤销后失败。
   - 如果 Cobo CAW 接入顺利，就用 CAW 做执行层；如果接入受阻，就用已经跑通的 Safe + Zodiac Roles fallback 证明同样的权限边界。

4. **Demo 是否容易理解**
   - Demo 应该避免只展示复杂链上细节，而是讲一个简单故事：
   - 用户让 Agent 购买一个白名单 API / 数据 / 工具服务。
   - Agent 生成付款建议。
   - Policy 检查预算、白名单、token、chain、金额和风险等级。
   - Cobo CAW 或 fallback 钱包只执行被允许的付款。
   - 所有结果都留下 audit log。

## Hackathon 推进

### 今日项目理解

今天我把 AgentScoope Wallet 的方向进一步收敛到 Cobo 赛道下的 Agent-Native Payments / Agent Resource Procurement。

这个项目的核心不是让 AI Agent 自由控制钱包，而是让 Agent 只能在用户预设的预算、白名单、时间窗口和策略边界里执行小额支付或资源采购。Agent 负责提出建议，Policy 负责判断是否允许，Cobo CAW 或 fallback 钱包执行层只执行被允许的动作。

我现在更清楚地把项目拆成三层：

1. **Agent 层**
   - 理解用户任务；
   - 生成付款 / 采购建议；
   - 不持有 owner 私钥；
   - 不直接决定是否付款。

2. **Policy 层**
   - 检查 recipient whitelist；
   - 检查 token / chain；
   - 检查单笔金额、日预算和 owner threshold；
   - 输出 `auto_pay_allowed`、`human_confirmation_required`、`blocked` 或 `owner_required`。

3. **Wallet / Execution 层**
   - 优先尝试 Cobo Agentic Wallet / Pact；
   - 如果 CAW 接入来不及，则用 Safe + Zodiac Roles fallback；
   - 执行成功、pending approval、拒绝原因和 tx hash 都进入 audit log。

### 今日项目一句话

AgentScoope Wallet 是一个面向 AI Agent 的受限执行钱包：它允许 Agent 在预算、白名单和策略边界内完成小额支付 / 资源采购，但不会让 Agent 获得无限钱包权限。

更短版本：

> Agent recommends. Policy decides. Cobo executes only if allowed.

中文版本：

> Agent 负责建议，Policy 负责裁决，Cobo 只执行被允许的付款。
