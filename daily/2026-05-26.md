- 不读取、不暴露 `.env`、私钥、seed phrase

- L2 / owner signature 级别不能由 agent 直接广播
- 所有执行 / 拒绝都需要留下 audit log

## Demo 测试结果

### Prompt B — App policy reject

测试内容：

```text
向 0x000000000000000000000000000000000000dEaD 转 0.5 USDC
```

结果：

```json
{
  "ok": false,
  "decision": "rejected",
  "reason": "transfer_to_unlisted_address",
  "rejectLayer": "app_policy",
  "humanCheckLevel": "L0"
}
```

结论：

- 非白名单地址被拒绝
- 未执行 `pay`
- 未广播交易
- audit 已记录 `tool_simulate`

### Prompt E — L2 owner required

测试内容：

```text
向白名单地址转 6 USDC
```

结果：

```json
{
  "ok": false,
  "decision": "rejected",
  "reason": "exceeds_max_per_tx",
  "rejectLayer": "app_policy",
  "humanCheckLevel": "L2"
}
```

结论：

- 该金额达到 L2 风险级别
- 未执行 `pay`
- 未广播交易
- 实际先命中 `maxPerTx = 1.0` 的应用策略限制，因此返回 `exceeds_max_per_tx`
- 安全拦截通过，但拒绝原因与 demo 文档中的 `requires_owner_signature` 不完全一致，原因是当前 policy 判断顺序中单笔上限先触发

## 今日收获

今天完成了 AgentScoope Wallet 从脚本 demo 到 Hermes Agent skill 的集成验证。通过 Prompt A–E 的测试，可以看到该钱包具备以下核心能力：

1. **白名单限制**：非白名单地址会在 app policy 层被拒绝。
2. **额度限制**：超过单笔额度的请求会被 `exceeds_max_per_tx` 拦截。
3. **模拟优先**：所有执行前都先进行 simulate，生成可读的交易摘要。
4. **L1 人工确认**：超过确认阈值但未达到 owner 签名阈值的交易，必须用户确认后才能执行。
5. **链上执行与审计**：成功交易会返回 Sepolia txHash，并写入 audit 日志。
6. **受限 Agent 执行模型**：Agent 不能任意花钱，只能在 Safe + Zodiac Roles + app policy 的多层限制下执行。

## Hackathon 推进

AgentScoope Wallet 今天从“概念 / 脚本 demo”推进到了“可被 Hermes Agent 调用并真实执行测试网交易”的 demo 阶段。

核心叙事更加清晰：

```text
User intent
  → Hermes Agent
  → agentscoope-wallet skill
  → get-policy / get-spending-status
  → simulate
  → risk classification: L0 / L1 / L2
  → human confirmation if needed
  → pay or reject
  → Safe + Zodiac Roles execution boundary
  → audit log + txHash
```

这个 demo 证明：AI Agent 可以执行链上支付，但它不能任意花钱，而是被 Safe、Zodiac Roles、app policy、额度限制、白名单、simulate、human confirm、audit log 多层约束。
