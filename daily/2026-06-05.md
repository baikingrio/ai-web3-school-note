## 今日学习主题：从“AI 炒币”收敛到“受限再平衡 Agent”

今天把项目主线定为 **PactTrader**，我对项目定位也更清楚了：

> PactTrader 不是让 AI 自由炒币，而是让 AI 在用户明确授权的 Pact / policy 边界内执行小额、可解释、可撤销、可审计的组合再平衡。

这个定位帮我避开了两个容易跑偏的方向：

1. **不做黑盒交易机器人**：不承诺收益率，不做复杂量化，也不让 Agent 直接控制钱包。
2. **不只做 AI 建议工具**：如果只生成交易建议，Web3 和 CAW 的价值体现不明显；关键是把建议接到受限执行和审计链路上。

所以今天的重点是把产品拆成一条清楚的链路：

```text
User Intent
  -> Portfolio Snapshot
  -> Strategy Agent / Rebalance Calculator
  -> TradeProposal JSON
  -> Local Risk Engine
  -> CAW Pact / Transaction Submit or Dry-run
  -> Allowed / Pending Approval / Denied
  -> SQLite Audit Log
  -> Dashboard Update
```

我现在更喜欢用这句话解释项目：

> Agent proposes. Policy decides. CAW executes only when allowed.

## 技术栈确认

今天确认的 PactTrader 技术栈：

- 前端：Nuxt.js、Vue、TypeScript、Tailwind CSS、shadcn-vue/ui。
- 执行层：Cobo Agentic Wallet（CAW）/ Pact。
- Agent / 策略层：Z.AI API + deterministic rebalance fallback。
- 数据库 / 日志：SQLite。
- 部署：Vercel。

我的理解是：

- **Nuxt / Vue** 负责把评委能看懂的 dashboard 做出来；
- **Z.AI API** 负责策略解释和结构化 proposal，但不能绕过风控；
- **本地 Risk Engine** 是第一道确定性硬检查；
- **CAW / Pact** 是执行层权限边界；
- **SQLite** 负责把 proposal、decision、execution result 和 audit event 留下来；
- **Vercel** 负责把 demo 变成可访问的线上展示。

今天对 PactTrader 的推进主要是“收敛”和“讲清楚”：

   从泛泛的 Agent 钱包 / 交易想法，收敛为一个更容易展示的场景：小额 USDC / WETH 组合再平衡。

2. **安全边界收敛**  
   用户先设置 token 白名单、协议白名单、单笔上限、日预算、滑点、止损和有效期。Agent 只能在这些条件内生成建议。

   MVP 不需要一开始就做复杂交易系统，而是先展示四条路径：

   - 未知 token / 非白名单协议被拒绝；
   - 审计日志可复盘。

4. **技术实现收敛**  
   用 Nuxt 做 dashboard，用 Z.AI API / fallback 做策略层，用 Risk Engine 和 CAW / Pact 展示执行边界，用 SQLite 留下审计记录。
