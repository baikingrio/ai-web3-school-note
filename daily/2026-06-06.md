## 今日学习主题：Agentic Wallet Demo 的稳定性边界

今天的一个重要体会是：Hackathon demo 不能只依赖“理想状态”。如果 Cobo API Key、Pact 权限、TSS Node 状态、远端钱包状态中任意一环不完整，前端就直接报错，评委看到的就不是安全链路，而是一个跑不起来的页面。

所以今天的工作重点是把 YieldAgent 的 demo 从“必须所有外部条件都完美”调整成“外部条件不完整时也能透明降级”：

```text
CAW CLI / TSS 状态
  -> 尽量解析本地可用信息
  -> 本地 TSS running 时不被 remote 403 误判为离线

Cobo Pact 提交
  -> 优先提交真实 Pact
  -> 权限不足时保存 local draft
  -> 不阻塞策略创建主流程

Cobo Balance API
  -> 优先查 Cobo 余额
  -> API Key 未配置时 fallback 到链上余额查询
  -> 钱包首页不再因为 COBO_NOT_CONFIGURED 直接 500
```

这让我更清楚地认识到：Agentic Wallet 项目里，“可控执行”不只是 policy 规则本身，也包括工程层面对失败状态的处理。一个好的 demo 应该诚实地区分真实 CAW 调用、local draft、dry-run 和 fallback，而不是把所有东西都包装成已经完全打通。

## 技术栈 / 架构复盘

今天继续沿用已确定的 YieldAgent 技术栈：

- 前端：Nuxt.js、Vue、TypeScript、Tailwind CSS、shadcn-vue/ui。
- 执行层：Cobo Agentic Wallet（CAW）/ Pact。
- Agent / 策略层：Z.AI API + deterministic fallback。
- 数据库 / 日志：SQLite。
- 部署：Vercel。

今天更具体地理解了几个模块的边界：

1. **CAW Readiness 不等于远端状态全部成功**  
   只要本地 TSS Node 真实 running，就不应该因为 remote 查询权限失败而把整个钱包准备状态判死。

2. **Pact 提交失败不等于策略创建失败**  
   YieldAgent 的策略创建可以先落成本地 Pact draft，并清楚告诉用户当前是 `local-draft`，等 CAW / Pact 权限恢复后再接真实提交。

3. **余额查询失败不等于钱包页面不可用**  
   Cobo Balance API 没配置时，页面仍然可以 fallback 到链上余额查询或展示 0，而不是直接让 `/api/wallet` 变成 500。

## Hackathon 推进

今天的推进不是“新增一个页面”，而是让 YieldAgent 的 MVP 更像一个可以演示的产品：

1. **创建 Agent Wallet 链路更稳**  
   避免本地 TSS 已经 running，却因为 remote status 权限问题被前端误判为不可用。

2. **创建策略链路更稳**  
   即使 Cobo Pact 权限还没完全打通，用户也能创建本地 Pact draft，看到 strategy -> pact -> audit 的主流程。

3. **钱包页面更稳**  
   没有 Cobo API Key 时不再直接崩溃，而是透明 fallback。

4. **Demo 叙事更清楚**  
   YieldAgent 可以明确展示四种状态：真实提交、local draft、dry-run、blocked / denied，而不是把所有失败都变成页面错误。
