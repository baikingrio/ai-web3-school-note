一个很直接的问题是：本地 SQLite 很方便，但 Vercel 的 `/tmp` 在 serverless 场景下并不稳定。策略、Pact、钱包准备状态、执行日志、Pact 执行凭证这些数据如果只存在临时文件里，Demo 就可能出现：

- 页面刷新后状态丢失；
- Pact 已经 active，但执行凭证找不到；
- 评审看到的 Dashboard 与真实 Cobo 状态不一致；
- 多个 serverless 实例之间状态不同步。

所以今天的核心学习是：**Agentic Wallet Demo 不只是要能跑通一条链路，还要让状态在部署环境里可恢复、可检查、可解释。**

## 今日学习主题：从本地 Demo 状态走向可部署持久化

今天同步到的代码里，最关键的方向是给 YieldAgent 增加 Supabase Postgres 持久化能力。我的理解是：

```text
本地开发：SQLite 足够快，适合调试
Vercel Demo：需要 Postgres / Supabase 保存关键状态
CAW / Pact：链上与 Cobo 状态是真实来源
Dashboard：要把本地状态、远端状态和错误提示对齐
```

这让我更清楚地看到 Web3 Agent 产品里“状态”的复杂性：

- 链上状态：资金、交易、协议仓位；
- Cobo 状态：Agent Wallet、Pact 审批、pact-scoped 执行凭证；
- 应用状态：策略表单、Dashboard 展示、日志、轮询状态；
- 部署状态：数据库连接、TSS 节点、Hermes runtime、API Key 是否配置正确。

如果这些状态只靠前端 mock 或临时文件串起来，Demo 看起来可能能跑，但很难解释问题出在哪里。今天的改动开始把这些状态拆清楚，并在部署自检里暴露 Postgres 连接状态。

主要改动方向：

1. **Supabase Postgres 持久化**
   - 增加 Postgres client、运行时状态、schema migration。
   - Repository 层支持从 SQLite 过渡到 Postgres。
   - Vercel 场景下不再只依赖 `/tmp` 临时 DB。

2. **部署自检增强**
   - Settings 页的 CAW deployment check 增加 Postgres 连接状态。
   - 后端增加 Postgres probe，能更早发现数据库配置问题。

3. **钱包地址与授权错误处理**
   - 加强 Agent Wallet EVM 地址解析和校验。
   - 钱包准备流程里对 pact-scoped key / 授权缺口的提示更清楚。
   - 避免把“地址没同步好”和“API Key 没权限”混成一个问题。

4. **Dashboard 与日志体验**
   - Dashboard 轮询和 log fetching 更稳定。
   - History 页和 Recent Logs 展示更接近真实执行 / pending / failed 状态。
   - Pact 管理流程里对执行、同步和日志更新的反馈更明确。

5. **策略创建体验**
   - 创建策略表单显示可用 USDC 余额。
   - 用户在输入预算前能看到 Agent Wallet 当前可用资金，减少超预算策略。

## Hackathon 推进

今天的进展对 YieldAgent Demo 很关键，因为它解决的是“评审打开公开链接时，状态是否稳定”的问题。

前几天更多是在证明：

> Agent 只能在 CAW / Pact 授权边界内执行。

今天进一步补上：

> 这些授权边界、执行日志和钱包状态，在 Vercel 部署后也应该可持久、可检查、可恢复。

这对 Demo 叙事很重要。评审不一定会关心我们用了 SQLite 还是 Postgres，但他们会关心：

- 页面是不是稳定；
- 钱包状态是不是可信；
- Pact 审批和执行结果是不是能追踪；
- 出错时是不是能看到明确原因，而不是只看到一个 502。

今天继续推进 YieldAgent，重点不是新增一个花哨功能，而是把 Demo 部署稳定性往前推进了一步。

我先同步了最新代码，今天主要看到几条线：一是增加 Supabase Postgres 持久化，让 Vercel 部署不再只依赖 `/tmp` 里的临时 SQLite；二是部署自检里加入 Postgres 连接状态，方便快速判断环境是不是配对；三是继续完善 Agent Wallet 地址校验、pact-scoped key 相关错误提示，以及 Dashboard / History 的日志展示。

今天最大的收获是：Web3 Agent Demo 不能只在本地跑通，还要能解释“状态从哪里来、凭证在哪里、执行日志怎么保存”。对 YieldAgent 来说，CAW / Pact 决定 Agent 能不能执行，数据库和日志决定用户能不能看懂发生了什么。Demo 稳定性其实也是安全叙事的一部分。

本地验证方面，我跑了 `pnpm test` 和 `pnpm build`，测试和生产构建都通过。下一步会继续关注线上环境变量、Postgres 连接、Cobo Pact 状态同步，以及 Demo 路径里的真实执行 / 拒绝日志是否足够清楚。
