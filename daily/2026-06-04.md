## 今日学习主题：合约不是“代码而已”，而是执行边界

今天我对合约的理解更具体了一点：

> 智能合约不是普通后端代码，而是部署到链上的规则。它一旦被调用，就会按照预先写好的逻辑执行，并留下可验证的链上结果。

在 AI Agent 场景里，合约尤其重要，因为 Agent 本身可能会出错、幻觉或被诱导。如果让 Agent 直接拥有无限制的钱包权限，风险会非常大。所以更合理的设计是：

1. **用户先定义边界**：哪些 token 可以用、哪些协议可以交互、单笔最多多少、每天最多多少、什么情况需要人工确认。
2. **Agent 只负责生成意图或建议**：例如“把一部分 USDC 放入某个收益策略”或“把组合调回目标比例”。
3. **Policy / Pact / 合约规则负责判断**：这个动作是否在用户授权范围内，是否超额，是否触碰黑名单或高风险协议。
4. **执行层只执行被允许的动作**：如果不符合规则，就应该被拒绝、暂停或进入人工确认。
5. **日志必须可复盘**：为什么允许、为什么拒绝、执行结果是什么，都要能解释清楚。

## 关键概念整理

### 1. EOA 与智能合约账户

- EOA 更像普通钱包地址，由私钥直接控制。
- 智能合约账户更像一个带规则的钱包，可以把权限、额度、白名单、多签、时间锁等逻辑写进去。
- 对 Agent 来说，智能合约账户更适合做受限执行，因为它可以把“不能做什么”写成确定性规则。

### 2. Approval 不是小事

以前我会把 `approve` 理解成一个普通授权动作，但今天复盘后意识到：

- 授权额度过大，可能让后续合约转走更多资产；
- 授权对象不可信，风险会被放大；
- 对 Agent 场景来说，approval 本身就应该被纳入 policy 检查。

所以在 Demo 里，授权不能只展示“成功”，还应该展示：授权给谁、授权多少、是否过期、是否可撤销。

### 3. Pact / Policy 是 Agent 的安全护栏

我现在更喜欢用一句话总结：

> Agent proposes. Policy decides. CAW executes only when allowed.

也就是说，AI 可以理解用户目标并生成 proposal，但最终是否执行，不能由 AI 自己说了算。尤其是涉及资金、授权、交易、收益策略时，必须有一层确定性的 policy 或 Pact 来约束。

### 4. 审计日志是产品功能，不只是数据库记录

合约学习也让我重新理解了日志的重要性。

在 Web2 产品里，日志通常是开发者排错用的；但在 AI × Web3 Agent 产品里，日志本身就是用户安全感的一部分。用户需要看到：

- Agent 提出了什么建议；
- 风险检查通过还是失败；
- 哪条规则触发了拒绝；
- 最终有没有提交到执行层；
- 如果执行了，交易 hash 或执行结果是什么。

这也是为什么 SQLite 日志在 MVP 里也值得保留：即使是本地 Demo，也要把“可解释、可追踪、可复盘”的链路讲清楚。
