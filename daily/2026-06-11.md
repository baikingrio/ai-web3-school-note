## WCB Learning

今天的学习重点不是做一个新页面，而是把 Web3 Agent 的执行边界讲清楚、写稳。

PactTrader 的核心安全模型是：

```text
Agent proposes.
Policy / Pact decides.
CAW executes only when allowed.
```

也就是说，Agent 不能拿到一个通用 API Key 后自由操作钱包。用户批准 Pact 之后，真正执行具体策略动作时，应该使用 Cobo 返回的 **pact-scoped 执行凭证**。这个凭证代表的是“这个 Pact 允许的那一小段权限”，而不是整个 Agent Wallet 的无限权限。

今天排查的线上 502，本质上就暴露了这个边界：代码在某些部署场景下缺少 Pact 子 Key 时，会 fallback 到 Agent 主 Key，导致 Cobo 侧拒绝执行。这个拒绝是合理的，说明权限模型在保护钱包；问题在于我们的应用层不应该 silent fallback，而应该明确提示缺少 active Pact 的执行凭证。

## 今日学习主题：Pact 执行不能把主 Key 当子 Key 用

今天最大的收获是：

> 对 Agentic Wallet 来说，“能读取 / 同步 Pact” 和 “能执行这个 Pact 下的交易”不是同一个权限。

可以简单理解为：

- **Agent 主 Key**：用于 Agent 侧创建钱包、提交 Pact、同步 Pact 状态等管理动作。
- **Pact 子 Key / pact-scoped key**：用户审批 Pact 后，Cobo 返回给该 Pact 的受限执行凭证。
- **active Pact 执行**：应该只用 Pact 子 Key，不应该在缺失时自动改用主 Key。

这个边界对 PactTrader 很重要，因为项目想表达的不是“AI 拿着钱包权限去交易”，而是“AI 在用户批准的 Pact 边界内做有限动作”。

## 技术栈 / 架构复盘

今天仍然围绕 PactTrader 当前技术栈：

- 前端 / 全栈框架：Nuxt.js / Vue / TypeScript。
- UI：Tailwind CSS / shadcn-vue。
- 执行层：Cobo Agentic Wallet / CAW / Pact。
- 状态与日志：SQLite。
- 部署：Vercel + 远端 Hermes / CAW runtime。

今天重点看的链路：

```text
/api/pacts/:id/execute
  -> resolvePactById
  -> refreshCoboPactStatus
  -> resolveExecutionCredentials
  -> executeFirstPactRecipe
  -> Cobo TransactionsApi / TransactionRecordsApi
```

关键修复点：

1. **active Cobo Pact 必须使用 pact-scoped key**  
   如果本地缓存里没有 Pact 子 Key，不能再 fallback 到 Agent 主 Key。

2. **缺 key 时先尝试刷新 Pact 详情**  
   Vercel 场景下 SQLite 可能在 `/tmp`，缓存不稳定。所以 `/execute` 缺少执行 key 时，应先尝试从 Cobo Pact 详情刷新，而不是直接进入错误执行。

3. **错误信息要说清真实问题**  
   之前错误文案容易让人以为“再配置 Agent 主 Key 就行”。今天改成明确提示：缺的是 active Pact 的 pact-scoped 执行凭证，也就是 Pact 子 Key。

4. **保留 `src_addr`**  
   Cobo contract call schema 要求显式传 `src_addr`。不传会触发参数校验错误，所以最终执行请求仍然需要带 Agent Wallet 地址；真正的问题不是 `src_addr`，而是执行凭证的授权范围。

## Hackathon 推进

今天的推进对 PactTrader Demo 很关键，因为它把“为什么这个 Agent 是安全的”落到了代码层。

如果评审问：

> Agent 会不会拿着钱包权限乱执行？

今天的答案可以更清楚：

- Agent 主 Key 不能被当作 active Pact 的执行凭证。
- 用户审批后返回的 Pact 子 Key 才能执行对应 Pact 范围内的动作。
- 如果 Pact 子 Key 缺失，系统应该停止并提示，而不是偷偷换一个更大的权限继续执行。
- 这和 PactTrader 的产品定位一致：AI 只是提出策略，真正执行必须经过 CAW / Pact 的受限授权边界。

这让 Demo 叙事从“能不能执行一笔交易”变成“如何证明 Agent 只能在授权边界内执行”。

## 今日打卡草稿

今天继续推进 PactTrader，重点排查的是 Cobo Pact 执行时的授权问题。表面上看是 `/execute` 接口返回 502，但深入看后发现，这不是普通前端问题，而是 active Pact 执行时 API Key 使用边界不清楚。

我今天把 Agent 主 Key 和 Pact 子 Key 的职责重新理了一遍：Agent 主 Key 更适合做钱包 / Pact 的管理和同步；真正执行一个已经 active 的 Pact 时，应该使用 Cobo 返回的 pact-scoped key，也就是这个 Pact 自己的受限执行凭证。之前代码在缺少 Pact 子 Key 时会 fallback 到主 Key，这会让 Cobo 拒绝执行，也容易让产品语义变得不清楚。

今天最终修复成：active Cobo Pact 如果没有 pact-scoped key，就不再自动用主 Key 继续执行，而是先尝试刷新 Pact credential；如果还是没有，就明确提示缺少 Pact 子 Key。同时补了回归测试，确认这条边界不会被后续改坏。

这个问题对 PactTrader 的 Demo 很重要，因为项目想表达的是“AI 在用户批准的 Pact 边界内执行”，不是“AI 拿着钱包权限自由操作”。今天的收获是：Web3 Agent 的安全感，不只来自页面说明，也必须落实到每一条 credential 和执行路径上。
