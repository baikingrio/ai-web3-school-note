## 今日学习主题：Z.AI 负责建议，Cobo 负责受限执行

今天我把 AgentScoope Wallet 的架构重新拆成三层：

```text
Z.AI Agent Layer
  -> 理解任务、分析证据、生成付款建议

Policy / Risk Layer
  -> 检查白名单、额度、预算、确认阈值

Cobo / Wallet Execution Layer
  -> 在授权边界内执行付款、审批、拒绝和审计
```

这个拆分让我更清楚地看到，一个真正可用的钱包 Agent 不能只依赖 LLM 的判断。LLM 可以帮助理解上下文，但不能成为最终权限来源。

关键原则是：

> AI Recommendation ≠ Final Payment Decision

中文理解是：

> AI 建议不等于最终付款决定。

这对 AgentScoope Wallet 很重要。比如 AI 可能认为某个贡献者完成了任务，建议支付 0.5 USDC，但最终仍然要检查：

- 收款地址是否在白名单内；
- token 是否允许；
- chain 是否允许；
- 单笔金额是否超限；
- 今日预算是否足够；
- 是否需要人工确认；
- 是否超过 owner / 多签级别阈值。

只有这些检查都通过，钱包层才应该执行。

## Hackathon / AgentScoope Wallet 启发

今天最大的启发是：AgentScoope Wallet 不应该被讲成“AI 自动帮你转账”，而应该讲成：

> AI Agent 可以提出链上付款建议，但钱包权限和执行边界必须由可验证的 Web3 基础设施来约束。

这个方向比单纯做一个付款 demo 更完整，因为它同时回答了三个问题：

1. **AI 能做什么？**  
   理解任务、总结证据、生成付款建议、解释为什么付款或拒绝。

2. **AI 不能做什么？**  
   不能拿私钥，不能绕过白名单，不能直接执行超额付款，不能用自然语言覆盖 policy。

3. **Web3 / 钱包层负责什么？**  
   负责账户、授权、额度、审批、链上执行、拒绝原因和审计记录。

后续 demo 可以重点展示这个安全边界，而不是只展示“交易成功”。尤其是拒绝场景也很重要，因为它证明 Agent 不是无限权限。

## 今日 check-in 草稿

今天我继续推进 Hackathon 项目 AgentScoope Wallet 的方向整理，重点分析了如何把本次赞助商 Cobo 和 Z.AI 的能力自然地放进项目里。

我今天的理解是：Z.AI 更适合做 Agent 的“理解和建议层”，比如读取 bounty 任务、分析完成证据、生成付款建议和解释风险；Cobo 更适合做“受限钱包执行层”，比如通过 Agentic Wallet / Pact / policy / audit log 来限制 Agent 的实际链上操作权限。

这让我进一步明确了一个原则：AI 的建议不能等于最终付款决定。即使 Z.AI 建议付款，也必须经过白名单、金额上限、每日预算、人工确认阈值和 owner 权限等硬性检查。只有检查通过，钱包层才可以执行；如果收款人不在白名单或金额超限，就应该拒绝或进入 owner approval。

今天的收获是把 AgentScoope Wallet 从“AI 钱包 demo”继续收敛成一个更安全的三层结构：Z.AI 负责理解任务和建议，Policy 负责裁决，Cobo / Wallet 层负责受限执行和审计。这个方向也更符合 AI × Web3 的核心问题：不是让 AI 拿到无限权限，而是让 AI 在明确边界内安全地执行。
