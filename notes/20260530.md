# 2026-05-30 学习笔记：Agent Profile 与受限钱包执行 Agent

## 1. 今天学习的问题

今天围绕 Week 2 的 `Agent Identity / Agent Profile` 任务，整理 AgentScoope Wallet Assistant 的身份、能力和边界。

核心问题是：

> 一个可以参与链上动作的钱包 Agent，应该如何描述自己？它能做什么、不能做什么、如何被调用、如何验证、失败后如何处理？

这件事看起来像“写项目介绍”，但实际更像是在定义 Agent 的安全边界。如果这个边界不清楚，后面做 tool calling、自动支付或链上执行时，就很容易变成“AI 什么都能做”的危险设计。

## 2. Agent Profile 需要回答什么

我今天理解到，一个 Agent Profile 至少要回答这些问题：

1. **Identity**：这个 Agent 是谁，由谁维护，服务什么用户。
2. **Capability**：它有哪些具体能力，而不是泛泛地说“可以帮忙”。
3. **Input / Output**：它接收什么输入，输出什么结果。
4. **Interface**：用户或其他系统如何调用它。
5. **Verification**：执行结果如何被外部验证。
6. **Failure Handling**：失败时如何拒绝、记录、恢复或要求人工确认。
7. **Safety Boundary**：哪些事情无论用户怎么说都不能自动执行。

对 AgentScoope Wallet 来说，这些问题都和资金安全、权限控制、审计记录有关。

## 3. AgentScoope Wallet Assistant 的定位

我把它定义为：

> AgentScoope Wallet Assistant 是一个受限钱包执行 Agent，用来帮助用户把链上操作意图转成可检查、可确认、可审计的执行计划。

它不是 Safe owner，也不是拥有完整钱包权限的自动化脚本。

更准确地说，它只是一个受限执行方：

- AI 负责理解用户目标；
- policy 负责检查预算、白名单、合约和方法；
- simulation 负责执行前预览；
- Safe / Zodiac Roles 负责钱包层约束；
- human confirmation 负责拦截中高风险动作；
- audit log 负责复盘和 proof。

## 4. 它能做什么

AgentScoope Wallet Assistant 可以做：

- 解析自然语言链上任务；
- 生成结构化执行计划；
- 检查预算、白名单、合约和方法；
- 调用 simulation / dry-run；
- 把 simulation 结果解释成人能看懂的摘要；
- 判断风险等级：L0 / L1 / L2 / L3；
- 在低风险测试网场景下执行受限交易；
- 记录 tx hash、拒绝原因、policy snapshot 和 audit log；
- 帮助用户整理 WCB / Hackathon proof。

## 5. 它不能做什么

它不能做：

- 不能接触私钥、助记词、API Key、RPC token 或 `.env`；
- 不能成为 Safe owner；
- 不能绕过预算、白名单和 policy；
- 不能自动执行无限 approve、`setApprovalForAll`、合约升级、治理投票等高风险动作；
- 不能向未知地址或非白名单地址自动转账；
- 不能自动修改 Safe module、role、预算或白名单；
- 不能替用户在主网执行真实资金操作。

这里最重要的一点是：

> 高风险动作不是“让 AI 再确认一下就可以”，而是必须进入人工确认、owner / multisig 确认，或者直接拒绝。

## 6. AI 和 Web3 的分工

今天的一个关键理解是：AI 和 Web3 在这个项目里的职责不同。

AI 适合做：

- 理解意图；
- 生成计划；
- 解释风险；
- 调用工具；
- 总结结果；
- 生成 proof。

Web3 / wallet 层适合做：

- 账户控制；
- 权限限制；
- 预算约束；
- 白名单；
- 链上执行；
- 公开验证；
- 撤销权限；
- 审计记录。

如果只靠 AI 的 prompt 约束资产操作，是不够安全的。真正的边界应该落在 Safe、Zodiac Roles、policy、simulation 和 human confirmation 上。

## 7. 对 Hackathon 项目的启发

这次 Agent Profile 可以直接变成 AgentScoope Wallet 的产品说明：

1. 用户不是把钱包交给 AI，而是给 Agent 一个受限角色。
2. Agent 的每次执行都必须先生成结构化计划。
3. 计划必须经过 policy 和 simulation。
4. 低风险动作可以自动执行。
5. 中高风险动作必须人工确认。
6. 违反硬规则的动作直接拒绝。
7. 所有执行和拒绝都要写入 audit log。

后续 demo 可以专门展示三类结果：

- **允许执行**：额度内、白名单内、simulation 通过。
- **需要确认**：接近预算上限、风险较高、涉及权限变化。
- **直接拒绝**：未知地址、未知合约、无限授权、请求私钥或 `.env`。

这样比只展示一笔成功交易更有说服力，因为它能证明系统真的有边界。
